# Universal2MD 格式转换器 PRD

## AI 速读卡

- 产品一句话：支持将 PDF, DOCX, HWP, WPS 等 13 种主流文档格式高质量转换为 Markdown 的本地优先转换工具。
- 核心循环：用户拖放多格式文档 -> 本地引擎按格式特征精准解析提取 -> 实时预览 Markdown 并连同媒体文件打包导出。
- 目标平台：桌面端（推荐 macOS 和 Windows 双平台客户端，例如 Tauri + Rust）结合核心 CLI。
- 硬约束：必须完全本地离线处理以保障数据隐私；必须输出标准 GitHub Flavored Markdown (GFM)；媒体文件必须被正确提取为本地引用。
- 推荐默认：技术栈使用 Tauri 作为桌面容器，Rust 负责底层多格式解析。
- 发挥空间：批量转换时的拖拽动效与进度可视化；针对断裂段落和乱码的正则自动修复；Markdown 预览的主题样式。
- P0 验收：导入包含标题、表格和图片的 DOCX，成功输出 md 文件及对应的 assets 文件夹。
- 最容易翻车：PDF 的多栏排版解析导致段落错位；WPS 和 HWP 闭源格式的解析失败；大文件处理导致内存溢出。
- 超预期机会：自动识别文档中的公式并转为 LaTeX 格式；为扫描版 PDF 接入轻量本地 OCR；为长文档自动生成目录。

## 第一章：产品概述

Universal2MD 是一款跨格式解析工具，让大模型研究者、知识管理爱好者和开发者能够将纷繁复杂的专有文档格式统一归档为清晰的 Markdown，而无需忍受错乱的排版、丢失的图片和上传云端带来的隐私风险。

#### 4. 高级格式与排版支持 (Advanced Formatting)
引擎需尽可能将原文档丰富的排版元素映射为对应的 Markdown 高级扩展语法，需支持以下语法集：
- **标题**：`# H1` 到 `###### H6`，以及 Setext 风格 `===` `---`
- **文本修饰**：粗体（`**` 或 `__`）、斜体（`*` 或 `_`）、粗斜体、删除线（`~~`）、高亮（`==高亮==`）
- **上下标**：下标（`~`）、上标（`^`）
- **代码**：行内代码（`` ` ``），代码块（` ``` ` 或 `~~~`）
- **链接与图片**：行内式、参考式（`[id]: url`）、带标题悬浮、图片嵌套链接（`[![alt](img)](link)`）
- **列表**：无序列表（`-`, `*`, `+`）、有序列表（`1.`）、嵌套列表、任务列表（`- [ ]`, `- [x]`）
- **引用块**：`>` 及多层嵌套引用
- **分隔线**：`---`, `***`, `___`
- **表格**：Pipe Tables（包括左对齐、居中、右对齐 `|:---|:---:|---:|`）
- **高级扩展**：
  - 脚注：`[^1]` 及 `[^1]: 脚注内容`
  - 定义列表：`术语 : 定义`
  - 缩写词：`*[缩写]: 全称`
  - Emoji 支持：`:smile:`, `:heart:`, `:+1:`
  - 数学公式：内联公式 `$E=mc^2$`，块级公式 `$$公式块$$`
  - 图表：支持 Mermaid 语法块 ` ```mermaid `
  - 目录：支持 `[TOC]` 占位符或生成结构
  - 属性注入：支持元素级 ID/Class/Style 语法 `{#id .class}`, `{.class}`, `{style="color:red"}`
  - 转义字符支持：`\* \# \\ \` \_ \{\} \+ \- \. \!` 等对比表

### 1.1 差异化对比表

| 功能 | 竞品 | 本产品 | 实现方式 |
|---|---|---|---|
| 隐私与本地化 | 需上传云端或依赖云端转换 | 完全离线处理 | 使用 Rust 本地解析库 |
| 资产剥离 | 仅输出纯文本，丢失图片 | 图文分离并自动关联 | 自动创建 assets 文件夹保存资源 |
| 小众格式支持 | 不支持 HWP 等格式 | 全面覆盖多种指定格式 | 集成专用解析库提供适配 |

### 1.2 三类用户画像

- 知识管理爱好者
  - 角色：Obsidian 和 Notion 深度用户
  - 核心目标：将收集的各类电子书和文档导入自己的知识库中。
  - 对现有工具最大的不满：转换后图片丢失、排版惨不忍睹，需要大量手动二次排版。
  - 让他们愿意切换的那一个功能：极致的图文混排还原和原生 Markdown 列表/引用支持。

- 大模型研究者
  - 角色：构建企业内部知识库或私有 RAG 系统的工程师。
  - 核心目标：清洗各种非结构化文档，提取出干净、带层级的文本供大模型处理。
  - 对现有工具最大的不满：PDF 多栏识别错乱，表格解析为一堆不可读的文本。
  - 让他们愿意切换的那一个功能：基于版面分析的 PDF 解析机制和精确的 Markdown 表格输出。

- 日常办公族
  - 角色：需要处理同事发来的 PPTX 和 WPS 文档的人。
  - 核心目标：快速提取文字和图片，复制到其他工作流。
  - 对现有工具最大的不满：没有对应软件打不开，或只能截图。
  - 让他们愿意切换的那一个功能：一键拖拽即可预览和复制纯文本及图片。

### 1.3 可行性边界

| 在范围内（及原因） | 明确排除在外（及原因） |
|---|---|
| 本地提取文本、表格、图片、基础层级结构（保障核心转换质量） | 完美还原全部视觉样式（Markdown 不支持且违背初衷） |
| 处理 docx, epub, pdf 等开放或成熟格式 | 云端同步、云端账号系统（坚持本地优先和纯工具属性） |
| HWP, WPS 等小众闭源格式的尽力解析 | 处理带强加密的电子书或受密码保护的 PDF（技术与法律风险） |

### 1.4 约束分层

| 硬约束 | 推荐默认 | 发挥空间 |
|---|---|---|
| 必须在本地离线环境下完成所有转换操作 | 使用 Tauri 构建以保证性能和极小打包体积 | 转换完成后的动画或音效提示 |
| 输出的文件必须是标准 Markdown | 左侧列表，右侧预览的经典分栏布局 | 智能忽略小于 1KB 的装饰性图片 |
| 图片等静态资源必须存放在同级目录 | 按照原文件名生成目录结构进行输出 | 代码块的语言类型猜测和语法高亮 |

## 第四章：超越竞品的差异化功能

### 第 4.1 节 一站式通吃小众与主流格式的完美图文分离

1. 竞品为何没有这个功能：多数在线工具依赖开源的单一解析器，不支持直接输出带本地图片引用的纯净文档；对本地化或电纸书闭源格式支持不佳且配置门槛极高。
2. 本产品如何实现：采用模块化的多引擎架构。对于标准文档利用现有成熟库；对于电子书提取内部结构并映射；对于复杂格式接入版面分析。
3. 交互流程：
```text
拖入十几种不同格式的文件混杂在一起 -> 点击一键转换 -> 所有文件在其源目录下生成同名的文件夹 -> 文件夹中包含独立的文本文件和资产目录。
```
4. 风险与应对：专有加密可能导致解析失败。应对方案：提示用户建议先另存为 DOCX 后重试。

```text
+-----------------------------------------------------------+
| UI 层                                                     |
| 负责：拖拽交互、任务队列管理、内容预览渲染                |
+-----------------------------------------------------------+
| IPC 通信层                                                |
| 负责：安全传递文件路径、推送进度事件                      |
+-----------------------------------------------------------+
| 核心转换层                                                |
| 负责：根据不同后缀路由给具体的解析策略，执行 I/O 操作     |

```

| 库名 | 用途 | 为何优于替代方案 | 大致包体积 |
|---|---|---|---|
| Tauri | 桌面端框架 | 相比 Electron 减小体积，且 Rust 性能极佳 | 15MB |
| pandoc | 核心解析后备 | 开源界最强转换利器，覆盖多种格式 | 50MB |
| pdf2json | PDF解析增强 | 专用 PDF 库更利于版面提取 | 未知 |

最大的架构风险是某些闭源格式没有跨平台库。
可替换技术原则：推荐 Tauri，如果对 Rust 不熟悉，可用 Electron 替代。不可变的是：解析过程绝对不能发送数据到云端，生成的文档必须剥离媒体文件。

## 第十一章：开发者交接说明

你，负责实现的 AI 编程助手：

### a) 实现顺序建议
优先搭建核心引擎桥接测试通吃 DOCX 的命令行，这解锁了核心能力；接着构建前端拖拽界面；最后集成并添加更多解析器。

### b) 最可能导致返工的三个决策
- 决策 1：如何解析小众格式。安全的默认选择是通过系统自带应用转换；什么信号提示你需要改变方向：如果在特定平台上无法处理，需在前端屏蔽。
- 决策 2：图片引用路径。安全的默认选择是使用相对路径；什么信号提示你需要改变方向：如果预览模块无法渲染，需替换为虚拟协议路径。
- 决策 3：大文件读取。安全的默认选择是流读取；什么信号提示你需要改变方向：如果进程崩溃，必须重构解析器。

### c) 哪里要严格，哪里可以灵活
导出必须严格执行图文分离并使用相对路径，这是核心约束不可妥协。前端技术栈可以灵活建议选择你最擅长的。

### d) 已知的未知项
此处未解决：WPS 格式由于封闭性，目前市面上缺乏完美的跨平台解析器，可能需要逆向工程库。

### e) 验收剧本
验收剧本 1：在开发环境拖入 DOCX 文件，点击转换，验证是否生成图片资源文件夹。
验收剧本 2：拖入一张 JPG 图像，验证系统是否拦截并提示不支持。
验收剧本 3：拖入一本书的 EPUB 文件，验证层级标题是否完整提取。
